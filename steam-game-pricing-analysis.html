SteamScope Pricing Analysis: Auditing and Rebuilding a Flawed Data Pipeline

Author

Shravani Jadhav

Introduction

This project analyses Steam game pricing patterns by auditing an existing data pipeline and rebuilding it using data analytics best practices. The analysis involves reverse-engineering a legacy cleaning process, identifying data quality issues, improving the workflow, and developing a regression model to understand factors influencing game prices.

A2: Reconstructing Avery’s Pipeline

student_clean <- raw |>
  
  # Step 1: Remove rows with zero price
  filter(price > 0) |>
  
  # Step 2: Remove rows with zero reviews
  filter(total_reviews > 0) |>
  
  # Step 3: Remove rows with NA values in metacritic or developer_bio
  filter(!is.na(metacritic_score), !is.na(developer_bio)) |>
  
  # Step 4: Parse release_date and extract components
  mutate(
    release_date  = dmy(release_date),
    release_day   = day(release_date),
    release_month = month(release_date, label = TRUE, abbr = TRUE),
    release_year  = year(release_date)
  ) |>
  
  # Step 5: Change datatype of release_month
  mutate(release_month = as.character(release_month)) |>
  
  # Step 6: Standardise developer name to title case
  mutate(developer = str_to_title(developer)) |>
  
  # Step 7: Remove columns not in legacy
  select(-name, -release_date, -developer_bio) |>
  
  # Step 8: Count developer titles (within cleaned dataset)
  add_count(developer, name = "dev_titles") |>
  
  # Step 9: Create pt_hrs (Avery divided by 100, not 60 — reproducing the bug)
  mutate(pt_hrs = avg_playtime_mins / 100) |>
  
  # Step 10: Create lpeak = log(peak_players), rounded to 2dp
  mutate(lpeak = round(log(peak_players), 2)) |>
  
  # Step 11: Round positive_ratio to integer
  mutate(positive_ratio = round(positive_ratio, 0) |> as.integer()) |>
  
  # Step 12: Cap price at $40, then round to 2 decimal places
  mutate(price = round(pmin(price, 40), 2)) |>
 
  
  # Step 13: Reorder columns to match legacy exactly
  select(appid, developer, release_day, release_month, release_year,
         genre, genre_code, price, total_reviews, positive_ratio,
         avg_playtime_mins, peak_players, metacritic_score, platforms,
         pt_hrs, lpeak, dev_titles)
  

view(student_clean)

Part B: The Critique & “Version 2.0” Fix

To: Managing Director, SteamScope Analytics
From: Shravani Jadhav
Re: Auditing and Fixing of Data Pipeline

After analyzing the data and auditing Avery’s pipeline, we found the bad decisions she took, while not all were wrong, it was important to fix the pipeline. We ran a diagnostic check to understand how Avery’s data manipulations actively violated linear regression assumptions. We then cleaned and rebuilt the dataset properly by fixing the errors made by Avery and created a clean pipeline. We also built a new V2 model and plot and concluded by interpreting the model.

B1: The Audit

After careful examination of the data wrangling process implemented by Avery, I have discovered six bad practices. Not all steps are necessarily incorrect, like removing games with zero prices and zero reviews, date parsing, capitalizing the developers names, and calculating the log of peak players are all reasonable choices. The six errors which break the rules of good data science are given below.

Error 1: Playtime divided by 100 instead of 60

Avery knowingly divided avg_playtime_mins by 100 instead of 60 for converting minutes into hours. This creates an error of unit conversion and goes against the data integrity principle – computed columns should match reality in their value. As a result, all the values of playtime in the dataset are understated by approximately 40%. Any model for pricing based on this data, as well as any report to a client, would draw wrong conclusions regarding the relationship between player engagement and price.

Error 2: Price arbitrarily capped at $40

By using the pmin() function to force all prices greater than $40 to the value of $40, Avery arbitrarily truncated the top end of the price distribution without explanation, contrary to the representativeness principle that data should capture reality rather than a arbitrary limit. Out of 752 games in the cleaned dataset, 600+ were artificially set to $40, leaving the dependent variable with essentially no variation. As a result, the regression was useless, and SteamScope underestimated consumer demand for premium games by a significant amount.

Error 3: Rows deleted based on missing metacritic_score

By using ‘na.rm=TRUE’ in filter criteria, Avery excluded all rows for which metacritic_score had a missing value. Over 430 paid games were removed from the data in this way. Most indie games do not receive any ratings on Metacritic, so this is not a data issue, it just means the indie games are less popular. By removing these rows systematic sampling bias is introduced, violating the principle of representativeness. Because of this, SteamScope failed to analyze those indie games that most needed their help.

Error 4: Rows deleted based on missing developer_bio

There was no need to delete the rows with developer_bio NA values as that column was unncessary and was anyways going to be dropped. This introduces further sampling bias with no business justification. So Avery did this on purpose to create confusion. However, whole studio and all its games could be left out altogether, messing up the distribution in the study.

Error 5: Kept genre_code as numeric

The problem here is that instead of converting this variable into a factor, Avery uses it as a numeric one. In other words, genre_code variable is treated linearly, which is incorrect since genres can be considered as categories and do not have a linear dependence. Consequently, this causes an additional error and a biased result.

Error 6: Converted release_month to character

Avery converted release_month into character using ‘as.character(release_month)’, destroying its ordered structure. This meant that months were sorted alphabetically instead of chronologically during visualization and plotting. Due to this, seasonal trends in pricing and release patterns became harder to interpret data lost is structure.

B2: The Diagnostic Check

library(ggResidpanel)

# Run residual diagnostic panel on the legacy model
resid_panel(legacy_model, plots = "all", smoother = TRUE)

The residual diagnosticsshows that the legacy model violates essential assumptions for linear regression. The Q-Q plot reveals a large deviation from the diagonal, especially on the right side, implying that the distribution of residuals is not normally distributed. This makes sense due to the $40 error, where any prices exceeding $40 would be rounded down to $40, which caused a spike in the residuals around the ceiling. In addition, a fan shape can be seen in the residual plot, indicating heteroscedasticity, thus breaking another essential assumption of linear regression models, mainly due to the inability to predict values beyond $40.

Avery’s data manipulations lead to these problems. The $40 price cap created a ceiling effect, which compressed the response variable and distorted the residual distribution and variance. Since the genre_code is a categoric variable but is modeled as a numeric one, the model used one regression line for an intrinsically categorical variable. As such, there will be structured misfit within the residuals, because genres with different price categories have been lumped up together to give rise to greater residual variances. Dividing by 100 rather than 60 results in a scaling problem of around 40%. Although there is no direct regression violation here, the pt_hrs coefficient becomes meaningless, making any business conclusion drawn from it invalid.

Therefore, game pricing is mainly dependent on factors like quality perception (positive reviews), popularity (peak players), distribution channels (platforms) and genre of the game. Whereas, total_reviews and playtime hours are not significant, meaning only engagement volume does not impact pricing decisions for developers.

Conclusion

This analysis successfully reconstructed SteamScope’s original pricing pipeline, identified critical data quality issues, and developed an improved Version 2.0 workflow following data analytics best practices. The audit demonstrated that several transformation decisions in the original pipeline reduced model reliability by introducing data distortion, sampling bias, and incorrect variable handling. After correcting these issues, the improved model provided a more reliable understanding of the factors associated with game pricing, increasing explanatory power from R² = 0.049 to R² = 0.310.

The findings suggest that pricing decisions are influenced by factors such as game genre, player engagement, review sentiment, and platform availability. However, the model still explains only part of the variation in pricing, indicating that additional factors such as brand strength, marketing strategy, and market positioning should be considered before making final pricing recommendations. Overall, this project highlights the importance of reproducible data pipelines, data quality validation, and statistically sound modelling in producing trustworthy business insights.
